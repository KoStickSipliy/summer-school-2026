## Фича 1. Список слотов

### Краткая суть

Главный экран авторизованной зоны, на котором клиент видит доступные прогулки SUP-клуба и начинает сценарий записи. По умолчанию показываются предстоящие слоты на ближайшие 7 дней. Из списка клиент переходит к деталям конкретного слота или открывает фильтры.

Экран-носитель фичи:

- `SCR-002` — список слотов

### Бизнес-смысл

Фича решает базовую продуктовую задачу: заменить ручную запись через мессенджеры и дать клиенту самостоятельный вход в бронирование.

Связанные бизнес-требования:

- `BR-1` — устранить ручную путаницу и двойные брони
- `BR-2` — дать клиентам возможность записываться самостоятельно
- `BR-7` — стартовать с минимально необходимого MVP
- `BR-9` — продукт мобильный, используется с телефона

### Функциональные требования

Основные требования:

- `FR-9` — показать клиенту список предстоящих слотов
- `FR-38` — поддержать фильтрацию списка слотов
- `FR-9a` — из списка должен быть путь в карточку слота
- `FR-44` — слот опирается на реальные данные маршрута и места встречи

Что фича обязана делать:

- Показывать слоты по умолчанию на ближайшие 7 дней.
- Отображать для каждого слота минимум:
  - дату и время старта
  - маршрут и тип маршрута
  - инструктора
  - цену
  - всего мест и свободно мест
- Сортировать слоты по времени старта: ближайшие выше.
- Показывать заполненные и отменённые слоты в списке, если фильтр “только свободные” не включён.
- Помечать недоступные слоты как “мест нет” и не давать перейти в запись через CTA.
- Поддерживать pull-to-refresh для актуализации данных.
- Давать перейти к карточке слота.
- Давать открыть bottom sheet фильтров.

### Нефункциональные требования

Для этой фичи наиболее важны:

- `NFR-1` — mobile-first, крупные элементы, высокий контраст
- `NFR-2` — короткий путь к записи
- `NFR-4` — мобильное приложение iOS/Android
- `NFR-6` / `NFR-21` — быстрый отклик списка
- `NFR-10` / `NFR-23` — доступность сервиса в рабочее время клуба
- `NFR-24` — допустим офлайн-просмотр ранее загруженного списка
- `NFR-25` — доступность интерфейса

Практический смысл NFR для реализации:

- Экран должен быть читаемым на солнце и с одной руки.
- Первичная загрузка должна давать быстрый визуальный отклик через skeleton/loading state.
- При плохой сети список не должен “ломаться”: нужны empty/error/offline-состояния.
- Ключевая информация в карточке слота не должна передаваться только цветом.

### User Story

- `US-2` — как клиент, я хочу видеть список предстоящих слотов прогулок, чтобы выбрать подходящую прогулку
- `US-3` — как клиент, я хочу фильтровать список слотов, чтобы быстро находить подходящий вариант
- `US-13` — как клиент, я хочу видеть ранее загруженные данные офлайн, чтобы ориентироваться при нестабильной связи

### Use Case

Основной use case:

- `UC-3` — просмотр и фильтрация списка слотов

Краткий сценарий:

1. Клиент открывает экран списка слотов.
2. Система загружает слоты на ближайшие 7 дней.
3. Клиент просматривает список.
4. При необходимости открывает фильтры.
5. Клиент выбирает подходящий слот.
6. Переходит в карточку слота.

Альтернативы и крайние случаи:

- `A1` — сброс фильтров возвращает дефолтную выдачу
- `A2` — можно скрыть занятые слоты через `only_available=true`
- `E1` — нет слотов под условия фильтра
- `E2` — в горизонте выдачи нет слотов вообще

### Рекомендации к реализации

Для клиента:

- Реализовать экран как read-only список с явным разделением состояний `Loading -> Content -> Empty -> Error` по `LOGIC-008`.
- Хранить applied filters отдельно от загруженных данных, чтобы не смешивать UI-состояние и ответ API.
- Использовать пагинацию из API, даже если в MVP виден только первый экран выдачи.
- Поддержать кеш последнего успешного списка для офлайн-режима с пометкой “данные могли устареть”.
- Для недоступных слотов дизейблить переход, а не только менять цвет.

Для API-интеграции:

- Использовать `listSlots` (`GET /slots`) как основной источник данных.
- Не хардкодить лимиты маршрутов, цены и доступность; брать их только из API.
- Считать сервер источником истины по доступности мест.

Для UX:

- Заголовок экрана, пустые состояния и ошибка должны быть устойчивыми и короткими.
- Индикатор активных фильтров лучше показывать числом, а не только цветом.

## Фича 2. Фильтры слотов

### Краткая суть

Bottom sheet, который позволяет сузить список прогулок по критериям клиента: дате, типу маршрута, инструктору и наличию свободных мест. Сам bottom sheet не показывает результаты, а лишь собирает параметры и передаёт их обратно на экран списка слотов.

Экран-носитель фичи:

- `BS-001` — фильтры

### Бизнес-смысл

Фича ускоряет путь от каталога к записи и уменьшает число лишних действий. Особенно важна в мобильном сценарии, где пользователь быстро ищет релевантную прогулку “здесь и сейчас”.

Связанные бизнес-требования:

- `BR-2` — клиент должен самостоятельно находить и выбирать прогулки
- `BR-7` — MVP должен быть сфокусирован на критичных сценариях
- `BR-9` — использование с телефона

### Функциональные требования

Основные требования:

- `FR-38` — фильтрация по дате/периоду, типу маршрута, инструктору, наличию свободных мест
- `FR-9` — фильтры применяются к списку слотов

Что фича обязана делать:

- Открывать форму фильтров поверх списка слотов.
- Предзаполнять форму текущими активными фильтрами.
- Поддерживать фильтр по датам:
  - быстрые пресеты
  - произвольный диапазон “с / по”
- Поддерживать фильтр по типу маршрута:
  - `novice`
  - `experienced`
- Поддерживать фильтр по инструктору из справочника.
- Поддерживать переключатель “только свободные”.
- Давать действие “применить”.
- Давать действие “сбросить”.
- Закрываться без применения при dismiss.
- Скрывать блок фильтра по инструктору, если справочник пуст.

Семантика фильтров:

- внутри группы значений — `OR`
- между группами — `AND`
- границы периода дат — включительные

### Нефункциональные требования

Критичные NFR:

- `NFR-1` — крупные кликабельные элементы
- `NFR-2` — минимальное количество действий до результата
- `NFR-6` / `NFR-21` — быстрый отклик при применении фильтров
- `NFR-24` — единый error/retry-подход
- `NFR-25` — доступность для screen reader и больших шрифтов
- `NFR-15` / `NFR-16` — не раздувать MVP лишней сложностью

Практические следствия:

- Шторка должна быть удобна одной рукой.
- Взаимодействие не должно зависеть только от цвета выбранного состояния.
- Ошибка загрузки справочника инструкторов не должна блокировать весь сценарий.

### User Story

- `US-3` — как клиент, я хочу фильтровать список слотов, чтобы быстро находить подходящий вариант

Дополнительно фича помогает выполнить:

- `US-2` — быстрее выбрать прогулку из списка

### Use Case

Основной use case:

- `UC-3` — просмотр и фильтрация списка слотов

Краткий сценарий:

1. Клиент открывает список слотов.
2. Нажимает “Фильтры”.
3. Выбирает нужные условия.
4. Нажимает “Применить”.
5. Система возвращает клиента на экран списка и обновляет выдачу.

Альтернативы:

- `A1` — сбросить фильтры
- `A2` — оставить только свободные слоты
- `A3` — комбинировать фильтры по правилам `OR/AND`
- `A4` — скрыть фильтр по инструктору, если справочник пуст

Ошибки:

- `E1` — нет слотов под выбранные условия

### Рекомендации к реализации

Для клиента:

- Делить состояние на:
  - `draftFilters` — то, что пользователь меняет в шторке
  - `appliedFilters` — то, что реально применено к списку
- Не перезапрашивать список на каждый тап по фильтру; запрашивать только по “Применить”.
- Загрузку справочника инструкторов делать отдельной неблокирующей операцией.
- При ошибке загрузки инструкторов оставлять остальные фильтры рабочими.
- После `dismiss` без подтверждения не менять `appliedFilters`.

Для API-интеграции:

- Использовать `listInstructors` для наполнения фильтра по инструкторам.
- Формировать query для `listSlots` строго из текущих `appliedFilters`.
- Не отправлять пустые фильтры, если это меняет дефолтное поведение API.

Для UX:

- Показать активные фильтры на экране списка после применения.
- Кнопку “Применить” держать фиксированной снизу.
- Для пустого результата использовать отдельный empty state с CTA “изменить фильтры” или “сбросить”.

## Фича 3. Экран успешной записи

### Краткая суть

Полноэкранное подтверждение успешного бронирования после `createBooking`. Экран показывает сводку брони, напоминает об офлайн-оплате и даёт два следующего действия: перейти в “Мои бронирования” или вернуться к списку прогулок.

Экран-носитель фичи:

- `BS-002` — подтверждение записи, но по факту это экран, а не bottom sheet

### Бизнес-смысл

Эта фича завершает главный сценарий записи и снимает тревогу пользователя: бронь действительно создана, условия понятны, следующий шаг очевиден.

Связанные бизнес-требования:

- `BR-2` — самостоятельная онлайн-запись
- `BR-8` — оплата офлайн, приложение лишь показывает цену и фиксирует запись
- `BR-7` — MVP должен закрывать критичный сценарий записи

### Функциональные требования

Ключевые требования:

- `FR-10` — создать запись на выбранный слот
- `FR-30` — показать цену и зафиксировать запись; оплата офлайн
- `FR-33` — после ценного действия можно запрашивать push-разрешение
- `FR-35a` — созданная запись должна быть видна в “Моих бронированиях”

Что фича обязана делать:

- Открыться только после успешного `createBooking`.
- Показать, что запись создана.
- Показать сводку брони:
  - дата и время старта
  - маршрут
  - инструктор
  - количество мест
  - разбивка “свои / прокатные доски”
  - итоговая цена
- Показать напоминание, что оплата производится на месте.
- Дать кнопку:
  - `Мои бронирования`
  - `Готово`
- Не выполнять повторный запрос на загрузку данных при открытии: использовать уже полученный объект брони.
- При первой успешной брони инициировать логику запроса push-разрешения.

Что фича не должна делать:

- Не должна повторно подтверждать успех через snackbar/toast.
- Не должна быть доступна по обычной навигации “сама по себе”.
- Не должна иметь автозакрытие по таймеру.

### Нефункциональные требования

Наиболее важны:

- `NFR-1` — экран должен быть простым и читаемым
- `NFR-2` — не увеличивать сценарий записи лишними шагами
- `NFR-6` / `NFR-21` — успех должен открываться быстро после брони
- `NFR-13` / `NFR-17` — push как канал напоминаний MVP
- `NFR-25` — доступность и понятные статусы
- `NFR-16` — без лишних “наворотов” сверх MVP

Практические следствия:

- Экран должен быть визуально спокойным, без перегрузки.
- Итоговая цена и ключевые параметры должны быть заметны сразу.
- Все действия должны быть явными: нет скрытого dismiss, нет неоднозначного выхода.

### User Story

- `US-5` — как клиент, я хочу записаться на выбранный слот, чтобы забронировать место на прогулке

Фактически экран успешной записи — это завершающий шаг этой истории:

- после подтверждения создаётся запись
- места уменьшаются
- запись появляется в “Моих бронированиях”

### Use Case

Основной use case:

- `UC-1` — запись на прогулку

Фрагмент сценария, относящийся к этой фиче:

1. Клиент проходит сценарий выбора слота и параметров брони.
2. Подтверждает запись.
3. Система создаёт бронь и уменьшает счётчики доступности.
4. Система показывает подтверждение с ценой и добавляет запись в “Мои бронирования”.

Важные альтернативы и ограничения:

- `E1`, `E2`, `E3`, `E5` из `UC-1` должны быть обработаны раньше, на экране оформления брони
- экран успеха показывается только при реальном успешном результате

### Рекомендации к реализации

Для клиента:

- Реализовать `BS-002` как отдельный экран навигации, а не как modal sheet.
- Передавать на него объект брони из ответа `createBooking`, а не перезапрашивать её по `id`.
- Хранить экран как одно стабильное `Content`-состояние без `Loading/Error` при нормальном сценарии.
- Кнопку “Мои бронирования” сделать primary CTA.
- Кнопку “Готово” вести на корневой список прогулок, очищая стек промежуточного оформления брони.

Для UX:

- Не дублировать успех snackbar’ом: сам экран уже является подтверждением.
- Отдельно и коротко показывать напоминание про офлайн-оплату.
- Итоговую цену брать из серверного `price_total`, не пересчитывать локально на этом экране.

Для push-логики:

- Использовать `LOGIC-007`: спрашивать разрешение только после первой успешной брони.
- Отказ в разрешении не должен блокировать выход с экрана и дальнейшую работу приложения.

Для API и архитектуры:

- Источник истины — ответ `createBooking`.
- На границе клиента держать явный DTO/domain mapping для `Booking`.
- Не смешивать логику успеха с обработкой ошибок записи: ошибки должны оставаться на экране оформления брони.
