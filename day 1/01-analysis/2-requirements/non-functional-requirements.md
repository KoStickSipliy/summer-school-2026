# Нефункциональные требования

## Общие принципы

Нефункциональные требования описывают качества решения, а не бизнес-сценарии. Они применяются к клиентскому приложению и API для него.

## Производительность

**NFR-001 (Приоритет: High).** Основные экраны приложения должны открываться без заметной задержки при штатной работе сети.

**NFR-002 (Приоритет: High).** Приложение должно обеспечивать быстрый переход от просмотра слотов к оформлению записи без избыточного числа экранов и лишних ожиданий.

**NFR-003 (Приоритет: High).** Повторная загрузка списков слотов и записей должна выполняться без блокировки интерфейса.

## Надёжность и согласованность данных

**NFR-004 (Приоритет: Critical).** Приложение должно корректно обрабатывать отказ бэкенда при отсутствии свободных мест.

**NFR-005 (Приоритет: Critical).** Приложение не должно самостоятельно считать себя источником истины для бронирования; итоговое состояние записи определяется ответом API.

**NFR-006 (Приоритет: High).** Приложение должно корректно отображать актуальный статус записи после получения изменений от бэкенда.

**NFR-007 (Приоритет: High).** При сетевой ошибке или ошибке API приложение должно показывать понятное сообщение и возможность повторить действие там, где это применимо.

## Удобство использования

**NFR-008 (Приоритет: Critical).** Интерфейс должен быть ориентирован на использование с телефона.

**NFR-009 (Приоритет: High).** Основные действия должны быть доступны без сложной навигации и без избыточного количества шагов.

**NFR-010 (Приоритет: High).** Тексты, статусы и ошибки должны быть понятными для обычного клиента без терминов внутренней системы.

## Доступность

**NFR-011 (Приоритет: High).** Интерфейс должен быть пригоден для использования на мобильных устройствах с различными размерами экранов.

**NFR-012 (Приоритет: High).** Элементы управления должны быть достаточно крупными для уверенного нажатия пальцем.

**NFR-013 (Приоритет: High).** Контраст текста и управляющих элементов должен обеспечивать читаемость в типичных условиях использования в мастерской.

## Совместимость и платформенные ограничения

**NFR-014 (Приоритет: Critical).** Приложение должно поддерживать клиентский сценарий на мобильных устройствах.

**NFR-015 (Приоритет: Critical).** Приложение должно работать в связке с существующим бэкендом через API-контракт.

**NFR-016 (Приоритет: Critical).** Изменения в интерфейсах мастера и администратора не должны требоваться для текущей поставки.

## Безопасность

**NFR-017 (Приоритет: High).** Доступ к персональным данным клиента должен быть ограничен авторизованным пользователем.

**NFR-018 (Приоритет: High).** Приложение должно использовать только данные, полученные через API, и не хранить лишнюю чувствительную информацию без необходимости.

## Поддерживаемость

**NFR-019 (Приоритет: Medium).** Функции записи, отмены, уведомлений и оценки должны быть отделены друг от друга на уровне логики приложения.

**NFR-020 (Приоритет: Medium).** Коды статусов, ошибок и сообщений должны быть согласованы с API-контрактом и не должны зависеть от ручной интерпретации пользователем.
