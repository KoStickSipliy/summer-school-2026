# Описание домена и ограничения

> Этап 1. Артефакт 1 проекта: согласованное описание домена + список ограничений.
> Заполняется после ответов заказчика ([customer-questions.md](customer-questions.md)).

## Описание домена

Гончарная мастерская проводит платные групповые мастер-классы для клиентов. Основной сценарий домена — просмотр доступных слотов, выбор занятия, оформление записи на себя, получение напоминаний, отмена записи в допустимый срок, а также просмотр статуса посещения и оценка мастера после занятия.

Занятия имеют дату и время начала, программу, мастера, доступное количество мест и цену. Клиент видит расписание на ближайшие 7 дней по умолчанию и может уточнять выбор через фильтр по датам. При записи клиент может сразу выбрать, нужны ли ему собственные инструменты и фартук или нужен прокат от мастерской.

После подтверждения записи клиент получает уведомление. Перед началом занятия ему отправляются напоминания, а в случае отмены занятия со стороны мастерской клиент получает push-уведомление и сообщение внутри приложения. Если слот отменён мастерской, повторная запись на него запрещена.

После посещения занятия клиент может поставить мастеру оценку по пятибалльной шкале. Программы лояльности в первой версии нет, поэтому поведение системы в этом направлении ограничивается только базовым отображением данных клиента и его записей.


## Пользователи и роли

**Клиент**

Основной пользователь текущей поставки. Использует мобильное приложение, чтобы:
- просматривать слоты и фильтровать их по датам;
- записываться только на себя;
- выбирать дополнительные параметры записи, связанные с прокатом инструментов и фартука;
- отменять запись не позднее чем за 1 час до начала занятия;
- получать уведомления и напоминания;
- видеть отмены со стороны мастерской;
- оценивать мастера после занятия.

**Мастерская / администратор / владелец**

Это внешняя для текущей поставки роль. Она уже существует в инфраструктуре и не входит в разрабатываемый клиентский интерфейс. Именно со стороны мастерской формируются расписание, слоты, программы, сведения о мастерах и действия по отмене занятий.

**Мастер**

Также является внешней ролью существующей инфраструктуры. В рамках текущей поставки мастер не получает отдельного интерфейса, но его данные отображаются клиенту как часть информации о слоте и о занятии.


## Ограничения

**Бизнес-ограничения**

- Клиент в текущей версии записывается только на себя, без добавления сопровождающих.
- На старте доступны только групповые мастер-классы с уже сформированным расписанием.
- По умолчанию клиент видит слоты только на ближайшие 7 дней.
- Запись возможна только на свободные слоты, а вместимость слота определяется расписанием и правилами мастерской.
- Для новичковых групп действует ограничение по числу мест: не более 6 человек.
- Отменить запись можно только не позднее чем за 1 час до начала занятия.
- Если клиент пытается отменить запись позже, система должна показать, что отмена уже недоступна.
- Если занятие отменяет мастерская, запись сохраняется в статусе отмены от мастерской, а повторная запись на этот слот запрещена.
- Клиент должен получать напоминания и уведомления о записи, предстоящем занятии и отмене со стороны мастерской.
- На старте не используется онлайн-оплата, доступны только наличные или перевод на карту.
- Стоимость записи и выбранных опций должна быть показана клиенту до подтверждения бронирования.
- После занятия клиент может поставить мастеру только оценку по пятибалльной шкале.
- Программа лояльности не входит в первую версию.

**Технические / платформенные**

- В текущей поставке разрабатывается только клиентское мобильное приложение и API для него.
- Интерфейсы мастера и администратора уже существуют и не входят в текущий scope.
- Слоты, программы и мастера приходят из существующего бэкенда через API, приложение их не создаёт и не редактирует.
- Бэкенд считается black-box источником истины, а защита от двойного бронирования обеспечивается на его стороне.
- Клиентское приложение должно корректно обрабатывать отказ при попытке записаться на уже занятый слот.
- Основной сценарий использования ориентирован на телефон.
- Для уведомлений используются push-уведомления и сообщения внутри приложения.
- Исторических данных и легаси-схемы нет, поэтому каноническим источником структуры данных считается контракт API.
- При отмене занятия мастерской клиентское приложение должно отображать изменённый статус записи и причину отмены, если она приходит в API.
